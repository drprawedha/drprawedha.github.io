```html Building a Lightweight Hosting Platform with openSUSE 16 + Cockpit

Preambule

This project started as an experiment to explore openSUSE 16 as an alternative lightweight hosting platform. Instead of using a traditional control panel, the goal was to combine Cockpit, Nginx, PHP-FPM and MariaDB behind an existing reverse proxy infrastructure.

Architecture Overview

Internet
    │
    ▼
192.168.1.200
Ubuntu + Nginx + SSL
(Reverse Proxy)
    │
    ├── 192.168.1.201
    │      HestiaCP
    │
    └── 192.168.1.50
           openSUSE 16
           + Cockpit
           + Nginx
           + PHP-FPM
           + MariaDB
           + SELinux

SSL termination was handled by the Ubuntu reverse proxy while the openSUSE server only provided internal HTTP services.

Initial Setup

  • Installed openSUSE 16.
  • Installed Cockpit for administration.
  • Configured Nginx.
  • Installed PHP 8.4 and PHP-FPM.
  • Installed MariaDB.
  • Enabled Firewalld and Fail2ban.
  • Prepared WordPress deployment environment.

First Problems Encountered

  • 403 Forbidden after deploying WordPress.
  • 502 Bad Gateway from PHP-FPM.
  • SELinux denied access to files.
  • WordPress installer appeared without CSS.
  • Mixed Content errors due to reverse proxy.
  • HTTP 500 caused by malformed wp-config.php.

Virtual Host Discovery

Initially, nginx virtual hosts were placed inside: 

/etc/nginx/conf.d

However, openSUSE uses: 

/etc/nginx/vhosts.d

Moving the configuration to the correct directory solved the 403 Forbidden problem.

PHP-FPM Troubleshooting

The first configuration assumed PHP-FPM used a Unix socket: 

/run/php-fpm/www.sock

After inspection, openSUSE 16 was configured to listen on: 

127.0.0.1:9000

Updating nginx to use the correct backend resolved the 502 Bad Gateway issue.

SELinux Challenges

WordPress files extracted from temporary directories inherited incorrect labels: 

user_tmp_t

This caused multiple access denied errors.

The issue was resolved using: 

restorecon -RFv /srv/www
setsebool -P httpd_can_network_connect on

Mixed Content and Reverse Proxy

WordPress installer loaded successfully but appeared without CSS. Browser Developer Tools reported: 

Mixed Content

Since HTTPS terminated at the reverse proxy, WordPress generated HTTP URLs internally.

Additional FastCGI parameters were added: 

fastcgi_param HTTPS on;
fastcgi_param SERVER_PORT 443;
fastcgi_param HTTP_X_FORWARDED_PROTO https;

After forwarding HTTPS variables properly, the installer loaded normally.

Final Status

  • openSUSE 16 server successfully deployed.
  • Nginx and PHP-FPM operating correctly.
  • MariaDB configured.
  • SELinux working properly.
  • Cockpit administration available.
  • WordPress successfully installed.
  • HTTPS handled through reverse proxy.
  • rhesaprawedha.my.id fully operational.

Future Plans

The next objective is to transform Cockpit into a lightweight hosting panel called: 

oS-Hosting

Planned features include:

  • Domain Management
  • Database Management
  • WordPress Installer
  • Backup and Restore
  • SSL Management
  • Multi-user Hosting
  • Role Based Access

Lessons Learned

  • Always verify nginx directory conventions on different distributions.
  • Never assume PHP-FPM socket locations.
  • SELinux contexts are as important as file permissions.
  • Browser Developer Tools reveal problems faster than logs.
  • Reverse proxy deployments require proper HTTPS forwarding.
  • Understanding the underlying stack is more valuable than depending solely on a control panel.
```
