Preamble

This project was initiated to build a centralized and easy-to-maintain Mercure Solo Email Signature Generator. The main objective was to eliminate manual signature editing while ensuring consistent branding across Microsoft Outlook Desktop and Outlook Web.

Instead of distributing HTML signatures individually, the project uses a dynamic employee database and a reusable signature template that automatically generates personalized signatures based on URL parameters.

Project Objectives

  • Create a single reusable HTML email signature.
  • Separate employee data from presentation layer.
  • Provide consistent branding for every employee.
  • Support Outlook Desktop and Outlook Web.
  • Host assets centrally on the Mercure Solo website.
  • Simplify future maintenance.

Employee Database

Rather than hardcoding employee information inside HTML, all personal information is maintained inside employees.js.

Each employee record contains:

  • Full Name
  • Job Title
  • Hotel
  • Address
  • City
  • Phone Number
  • Email Address

Adding a new employee only requires creating another object inside employees.js without modifying the HTML template.

Dynamic URL Parameter

The signature page supports URL parameters to determine which employee information should be rendered. 

    index.html?user=rhesa
    index.html?user=dodit
    index.html?user=weni
    index.html?user=emma

JavaScript automatically loads the matching employee profile and updates the signature contents without requiring multiple HTML files.

Dynamic Rendering

JavaScript updates the following placeholders dynamically:

  • Name
  • Job Title
  • Hotel Name
  • Address
  • City
  • Phone Number
  • Email Address

This allows a single HTML template to be reused by every employee.

Brand Assets

All branding assets are centrally hosted to guarantee consistency across all generated signatures.

  • Mercure Banner
  • ALL Logo
  • Facebook Icon
  • Instagram Icon
  • LinkedIn Icon
  • YouTube Icon

Because all images are referenced by absolute URLs, updating an asset automatically updates every generated signature.

Outlook Compatibility Challenges

  • Outlook Desktop uses Microsoft Word rendering engine.
  • External CSS is often ignored after Copy & Paste.
  • Images may lose proportions.
  • Banner width may expand unexpectedly.
  • Table layouts behave differently between browsers and Outlook.

Solutions Implemented

  • Used HTML table layout for maximum compatibility.
  • Hosted all images on HTTPS.
  • Added explicit width and height attributes to images.
  • Prepared inline-friendly HTML structure.
  • Separated employee data from presentation.
  • Created reusable dynamic signature template.

Testing

The generated signature was tested in several environments:

  • Google Chrome
  • Microsoft Edge
  • Mozilla Firefox
  • Outlook Web
  • Microsoft Outlook Desktop

Testing focused on typography, spacing, image rendering, hyperlinks, and responsive behavior after copy-paste.

Lessons Learned

  • Separate data from presentation whenever possible.
  • HTML email development differs significantly from standard web development.
  • Outlook rendering requires careful table-based layouts.
  • Centralized assets simplify long-term maintenance.
  • Dynamic templates reduce duplication and improve scalability.
  • Reusable components make future enhancements significantly easier.
