Preambule

While validating a new WordPress deployment using the HestiaCP Application Installer, an unexpected error appeared: 

sh: 1: /usr/sbin/sendmail: not found
Error: Connection to localhost failed

Although the web server and MariaDB service were running normally, WordPress installation could not proceed. What started as a simple test eventually became a full troubleshooting session involving MariaDB, HestiaCP, reverse proxy configuration, SSL, and WordPress HTTPS handling.

Architecture Overview

Internet
    │
    ▼
Ubuntu + Nginx + SSL
(Reverse Proxy)
    │
    ▼
Ubuntu + HestiaCP
    + Nginx
    + Apache
    + PHP-FPM
    + MariaDB
    + WordPress

SSL termination is handled by the reverse proxy while the backend server provides internal HTTP services.

First Problems Encountered

  • WordPress installation failed.
  • Connection to localhost failed.
  • sendmail not found.
  • Nginx conflicting server_name warning.
  • SSL certificate errors.
  • WordPress installer loaded without CSS.
  • Mixed Content issues.
  • Reverse proxy HTTPS detection problems.

Investigating MariaDB

The first suspicion was that MariaDB itself had failed. However, the service was healthy and running correctly.

Attention shifted toward HestiaCP configuration. Several files were inspected: 

/root/.my.cnf
/usr/local/hestia/conf/.mysql.localhost
/usr/local/hestia/conf/mysql.conf

Eventually it was discovered that HestiaCP still contained outdated database connection settings.

Fixing HestiaCP Database Connectivity

After synchronizing HestiaCP database configuration with MariaDB, database provisioning became functional again.

Testing with the command: 

v-add-database

confirmed that database creation worked normally.

Nginx Server Name Conflict

While testing the reverse proxy, nginx reported: 

conflicting server name

Multiple virtual hosts were serving the same hostname. Removing duplicate server blocks solved the problem.

SSL Configuration

After cleaning nginx configuration, HTTPS setup still failed because certificates were missing.

Using Certbot generated the required certificates and HTTPS became available.

Reverse Proxy Configuration

Ubuntu nginx acts as the SSL endpoint and reverse proxy. The backend receives forwarded headers including:

  • Host
  • X-Forwarded-Proto
  • X-Forwarded-Host
  • X-Forwarded-For
  • HTTPS information

This allows backend applications to reconstruct the original request.

WordPress Without CSS

After WordPress installation succeeded, the dashboard loaded without stylesheets.

Only plain HTML was displayed.

Browser Developer Tools revealed mixed content issues. WordPress was generating HTTP URLs because HTTPS terminated at the reverse proxy.

Temporary Solution

A reverse-proxy detection snippet was added to: 

wp-config.php

Once HTTPS forwarding was recognized, WordPress assets and CSS loaded normally.

Permanent Solution

Editing every wp-config.php file is not scalable. Instead, Hestia templates were modified. 

/usr/local/hestia/data/templates/web/nginx/php-fpm/default.tpl

/usr/local/hestia/data/templates/web/nginx/php-fpm/default.stpl

Additional FastCGI parameters were introduced so future WordPress installations automatically recognize HTTPS behind reverse proxy.

Final Status

  • Reverse proxy operational.
  • SSL working properly.
  • MariaDB healthy.
  • HestiaCP database connectivity restored.
  • Automatic database creation functional.
  • WordPress installation successful.
  • CSS and assets loaded correctly.
  • HTTPS detection fixed.
  • Future deployments simplified.

Lessons Learned

  • Always verify database credentials stored by control panels.
  • Reverse proxy deployments require proper HTTPS forwarding.
  • Browser Developer Tools reveal problems faster than logs.
  • Duplicate nginx server blocks create unexpected behavior.
  • Control panels are convenient, but understanding the stack remains essential.
  • Template customization is preferable to repetitive manual fixes.
